Challenge 1: Ghost Admin

100 points Solved

Challenge Description

"In many situations you don't need signatures, the person's appearance is enough."

Analysis & Approach

We examined challenge-1/public_source.rs and found the following validation struct and instruction:

pub fn admin_withdraw(ctx: Context<AdminWithdraw>) -> Result<()> {
    require!(
        ctx.accounts.admin.key() == ctx.accounts.vault.admin,
        ErrorCode::Unauthorized
    );
            
    msg!("Withdrawal successful! ST_FLAG{$$$$$$$$$$$$$$$}");
    Ok(())
}

#[derive(Accounts)]
pub struct AdminWithdraw<'info> {
    pub admin: UncheckedAccount<'info>,

    #[account(
        seeds = [b"vault"],
        bump
    )]
    pub vault: Account<'info, Vault>,
}

Vulnerability: The admin account in AdminWithdraw is defined as UncheckedAccount. The program checks that the key matches the vault's registered admin key, but fails to assert that the admin account is actually a transaction signer (there is no Signer type enforcement or manual signature verification).

Exploit: We passed the public key of the vault's admin as the admin account in the transaction, signed it only with our solver wallet, and bypassed authentication successfully.

Retrieved Flag ST_FLAG{trust_n0_pubk3y}

Challenge 3: Logs of Truth

100 points Solved Program ID: 5zzgo53dmRCCwrxX3q7UDmssW26Gh4f7Y8J2mEE7Rvds

Challenge Description

"The truth is hidden in numbers, from SOL to lamport, but only the right ones unlock the secrets."

Analysis & Approach

We verified the transaction logs for the program and observed that successful invocations printed parts of a flag. The program threw input validation errors when inputs were out of bounds or not exact multiples. By parsing the exact transaction instruction data of past successful calls, we recovered three inputs:

  • Part 1 (logged ST_FLAG{1sol_): Input 1,000,000,000 lamports (1 SOL)
  • Part 2 (logged 2sol_3sol_): Input 2,000,000,000 lamports (2 SOL)
  • Part 3 (logged truth}): Input 3,000,000,000 lamports (3 SOL)

Calling the verifyNumber instruction with these three values in sequence allowed us to reconstruct the complete flag.

Challenge 12: Voucher Roulette

300 points Solved Program ID: Adom7b6AmcmaPPLsB5bz8KZYN5NfuqYWeHLQ9jYeWBbY

Challenge Description

"One cannot win without failures."

Analysis & Approach

The program contains a redeem_code instruction that compares the MD5 hash of character-index inputs against pre-defined values:

pub fn redeem_code(_ctx: Context<Redeem>, code: String) -> Result<()> {
    require!(code.len() == 17, ErrorCode::WrongLength);

    for (i, ch) in code.chars().enumerate() {
        let hash_input = format!("{}{}", ch, i);
        let hash = md5::compute(hash_input.as_bytes());
        let hash_u32s = hash_to_u32_array(hash.0);

        if hash_u32s != EXPECTED_HASHES[i] {
            return Err(get_flag_error(i).into());
        }
    }
}

Vulnerability: Because each character-position pair format!("{}{}", ch, i) is checked independently of other characters, it can be brute-forced position by position.

Solution: We wrote a simple loop cracking each character at indexes 0 to 16. Brute-forcing the 17 characters cracked the full flag code.

Retrieved Flag ST_FLAG{g00d_on3}

Challenge 14: Sus Protocol

400 points Solved Program ID: A7w5Zz1aycLmCNJ2qWsYg8scL4DD4wvFWcCbwhPjCLCg

Challenge Description

"While the setup is important, cleaning up properly is too."

Analysis & Approach

The program maintains a vector of collateral and loan positions for users. However, it uses take_while to query visible collateral/loans, but uses filter to evaluate the exploit trigger:

// Visible loans calculation
let total_loans: u64 = user_account
    .positions
    .iter()
    .take_while(|p| p.account != Pubkey::default())
    .filter(|p| matches!(p.position_type, PositionType::Loan))
    .map(|p| p.amount)
    .sum();

// Real/Actual loans calculation
let real_total_loans: u64 = user_account
    .positions
    .iter()
    .filter(|p| p.account != Pubkey::default())
    .filter(|p| matches!(p.position_type, PositionType::Loan))
    .map(|p| p.amount)
    .sum();

Vulnerability: If a position is closed, its account is set to Pubkey::default(). Since take_while terminates immediately upon seeing a default public key, any positions pushed after it are completely hidden from the lending calculation. This allows us to perform duplicate borrows against the same collateral.

Exploit Walkthrough:

  1. Deposit 0.2 SOL (Index 0).
  2. Deposit 0.2 SOL (Index 1).
  3. Withdraw 0.2 SOL from Position 1 (index 1), reducing its amount to 0.
  4. Close Position 1 (sets account to Pubkey::default()).
  5. Deposit 0.1 SOL (Index 2). This makes real_total_collateral = 0.3 SOL and visible_collateral = 0.2 SOL.
  6. Borrow 0.1 SOL three times. Because the loan positions are pushed to index 3+, they sit after the closed position and are invisible to take_while, bypassing the collateral validation check. This allows us to borrow 0.3 SOL total, triggering the exploit.
Retrieved Flag ST_FLAG{t00k_y0u_a_wh1le}
